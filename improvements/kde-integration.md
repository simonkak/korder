# KDE Integration Improvements

## Current touchpoints

Korder already lives well inside Plasma 6 — verified by reading the modules below:

- **PipeWire client identity** (`src/korder/app.py`): `setproctitle` + `PIPEWIRE_PROPS` / `PIPEWIRE_ALSA` / `PULSE_PROP` so the Plasma volume mixer shows "Korder" with the bundled icon, not "ALSA plug-in [python3.12]".
- **`KStatusNotifierItem` tray** (`src/korder/app.py::_make_tray`): waveform icon recolored per state (idle / wake-listening / dictating), tooltip + context menu, left-click toggle.
- **Layer-shell OSD** (`src/korder/ui/osd.py`, prototype in `plasma_osd/test_osd.qml`): `org.kde.layershell` overlay, focus-less, picks up KWin's Blur effect.
- **Auto-installed `.desktop` + hicolor SVG** (`_install_desktop_entry`, `_install_tray_icon_in_theme`): per-user, idempotent, written so PipeWire stream-→-app resolution and the launcher both find Korder.
- **MPRIS read + control** (`src/korder/audio/_mpris.py`, `src/korder/actions/now_playing.py`): `qdbus6` against `org.mpris.MediaPlayer2.*` for player enumeration, `xesam:` metadata, `PlaybackStatus`. Same wire pauses/resumes during TTS.
- **WirePlumber CLI for ducking + volume actions** (`src/korder/audio/ducker.py`, `actions/media.py`): `wpctl get-volume @DEFAULT_AUDIO_SINK@`, `set-volume`. Avoids the kernel-keycode race against KDE's volume cache.
- **Spotify D-Bus `OpenUri`** (`actions/spotify.py`): MPRIS `Player.OpenUri spotify:...`, with `xdg-open` fallback to launch the client.
- **Power & lock** (`actions/system.py`): `xdg-screensaver lock`, `systemctl poweroff/reboot/suspend` (each gated behind voice confirmation).
- **`xdg-open` URL actions** (`actions/web.py`) for web search, YouTube, Wikipedia, Maps.
- **`notify-send`** (`actions/now_playing.py`): one-line freedesktop notification with `-a Korder -i media-playback-start`.
- **`ydotool` (uinput)** (`inject.py`) for keystroke synthesis on Wayland — the only Wayland-friendly option on KDE today.
- **External KGlobalAccel binding** (README): user binds `korder toggle` / `korder cancel` as a Custom Shortcut. The Unix-socket IPC in `app.py::_start_ipc_server` is what those hotkey commands talk to.

## Recommendations

### Low-hanging fruit (≤4h, KDE-specific)

- **Register global shortcuts via `org.freedesktop.portal.GlobalShortcuts`** — eliminates the README step where users hand-bind `korder toggle` and `korder cancel` in System Settings. The xdg-desktop-portal-kde implementation backs it with KGlobalAccel; first run pops a one-shot Plasma dialog asking the user to confirm/edit the chord (`Ctrl+Space` / `Esc` defaults), then it persists with the same UX as any KDE-native shortcut. Talks to portal D-Bus `org.freedesktop.portal.GlobalShortcuts` (object `/org/freedesktop/portal/desktop`); fallback is the existing socket commands so multi-user setups without the portal still work. Wayland-native — actually the *only* sanctioned global-hotkey path on Wayland; the current "Custom Shortcut → Command/URL" workaround is a side-channel.

- **Use `org.freedesktop.Notifications` directly instead of `notify-send`** — switch `now_playing._notify` and the rest of the spawn-`notify-send` sites to `QDBusInterface` calls (PySide6 already imports the rest of Qt). Wins: progress updates by replacing a notification ID instead of stacking new ones (e.g. "Searching → Found → Playing" Spotify lifecycle becomes a single self-updating bubble); action buttons on the bubble ("Show track on Spotify"); per-stream sound suppression via `suppress-sound` hint when TTS is also speaking the same line. Same D-Bus service; zero new deps.

- **Quit/cancel signals to the existing instance via the session bus** — `xdg-open`-style "is it running?" polling currently goes through the bespoke Unix socket. Publishing `org.kde.korder` (or `org.freedesktop.Application`-compliant `local.korder.Korder`) on the session bus lets `kdialog`, scripts, `dbus-send` invocations, and KRunner cleanly drive the running app. Costs ~30 lines around `_start_ipc_server`. Keep the socket as a fallback for non-D-Bus invocations.

- **`StatusNotifierItem` overflow menu polish** — the tray menu today is static. KSNI supports per-state submenus and "attention" icons. Two cheap wins: (a) flip to a pulsing `org.kde.attention-` icon while a long Whisper transcription is running (eyes-busy users learn the OSD is mid-something), (b) add a *Wake phrase: hey_jarvis ▸* submenu listing the openWakeWord catalog so the user can switch phrases without opening Settings. All Qt API on the existing `QSystemTrayIcon`.

- **MPRIS server, not just MPRIS client** — Korder reads MPRIS metadata. It can also *expose* an MPRIS player while TTS is mid-utterance: title="Korder is speaking…", PlaybackStatus="Playing", with `Pause` mapped to "shut up", `Stop` mapped to "cancel session", position+length tracked from the synthesizer. The Plasma media widget then shows Korder alongside Spotify, and headphone media keys interact with it. Service name `org.mpris.MediaPlayer2.korder`, interface `org.mpris.MediaPlayer2.Player` — same protocol Korder already reads.

- **`kwriteconfig6` / `kreadconfig6` for "set Plasma dark mode" / "switch wallpaper"** — small surface, all CLI: `plasma-apply-colorscheme BreezeDark`, `plasma-apply-lookandfeel`, `plasma-apply-wallpaperimage`. Wraps cleanly into actions `set_dark_mode`, `set_light_mode`, `set_wallpaper <path>`. No D-Bus glue — these CLIs are part of `plasma-workspace`. Caveat: the wallpaper one is per-screen, query screens via `qdbus6 org.kde.plasmashell /PlasmaShell evaluateScript` if needed.

- **`spectacle --background` voice action for screenshots** — Spectacle ships with KDE and accepts `--region`, `--activewindow`, `--fullscreen`, `--copy-image`. Three small registered actions ("screenshot region", "screenshot active window", "screenshot screen"). No D-Bus, just a binary. Plays well with the existing now_playing-style notify-send confirmation: include the saved path, click to open in Gwenview.

- **systemd user unit** — currently absent. Ship a `korder.service` template under `~/.config/systemd/user/korder.service` (auto-installed alongside the `.desktop` entry, mirror its idempotency pattern in `_install_desktop_entry`). Lets users `systemctl --user enable --now korder` for autostart instead of relying on the `.desktop` "Autostart" checkbox in System Settings. Useful sub-recommendation: `Restart=on-failure` so a Whisper crash doesn't permanently disable dictation. KDE's session manager *can* manage `.desktop` autostart, but the systemd path also gives journald logs out of the box (the new logging migration in `feat/logging-migration` benefits directly).

### Higher-value (1-3 days)

- **KWin script bridge for window-aware actions** — KWin exposes a JS scripting API via D-Bus (`org.kde.KWin /Scripting`) that can enumerate windows, query active client (`workspace.activeClient`), move/resize/maximize, and switch virtual desktops. A small Korder-side KWin script (loaded via `loadScript`/`run`) would unblock voice commands like "next desktop", "tile this left", "minimize all", "focus Firefox". Crucially, this is the missing piece for the parked `feat/at-spi-and-vision-actions` branch: AT-SPI doesn't enumerate windows on Wayland, but KWin scripting does — Korder can read the window list, ask Gemma which window the user means, and send focus/move ops back through the script. Voice commands enabled: *"focus Kate"*, *"close this window"*, *"send to next desktop"*, *"tile this to the right half"*, *"show all windows"* (`workspace.slotToggleShowDesktop()`), *"snap to screen 2"*. Wayland-native; no X11 fallback needed because KWin scripting *is* the Wayland-blessed window-management API.

- **KRunner runner plug-in** — Plasma's universal launcher (`Alt+Space`) accepts third-party runners over D-Bus (the `RunnerV2` protocol, service registered under `org.kde.kf6.krunner`). A "korder:" runner would let the user type a *fragment* in KRunner ("kor play linkin park") and have Korder dispatch the action without opening the mic. Useful for keyboard-first moments where the user knows the action but doesn't want to speak. Bidirectional value: Korder also gains an interactive debugging surface — every voice action becomes manually testable from KRunner. Implementation is one D-Bus service exposing `Match(query)` + `Run(matchId, actionId)`; the action registry already produces the right metadata (description + triggers). Keep it bounded to non-destructive actions or carry the same confirmation-word gate as voice.

- **KDED module migration (consider, then probably skip — keep as standalone)** — KDED is KDE's daemon host; modules load on session start, share KDED's process, and get cheap KGlobalAccel + KIO + DBus integration. *On paper* a good fit because Korder is "always-on background service with tray". *In practice* the migration cost is high (rewrite to KCoreAddons module convention, lose Python — KDED modules are C++/Qt) and the wins are modest because the same DBus/portal/systemd APIs are reachable from a regular process. Recommendation: stay standalone, but adopt a `ksvc.korder` D-Bus name on the session bus so other Plasma services can talk to Korder as if it *were* a KDED module. This is a higher-value entry because the analysis itself is a deliverable — anyone proposing "make Korder a KDED module" should read this first.

- **KDE Connect outbound** — KDE Connect exposes per-device D-Bus services under `org.kde.kdeconnect` (`/modules/kdeconnect/devices/<id>/...`). Korder could add device-targeted actions: *"send my last clipboard to my phone"* (`requestSendNotification`/`shareUrl`), *"ring my phone"* (`findMyPhone`), *"text Mum I'll be late"* (`sendSms` via `/modules/kdeconnect/devices/<id>/sms`), *"play next on phone"* (the phone-side MPRIS proxy). The user is the only consumer, so the security-model conversation is bounded — KDE Connect is already paired and authenticated. Wayland-clean (no input synthesis). The voice commands look like Spotify-style parameterized actions: extract `device_name` + `body` via Gemma, fire D-Bus call. Day 1: enumerate paired devices and add `kdeconnect_ring`. Day 2: SMS. Day 3: clipboard share + media-control proxy.

- **Plasma Activities-aware action vocabulary** — `org.kde.ActivityManager` exposes the current Activity ID (e.g., "Work", "Personal", "Gaming"). Korder could swap which actions Gemma sees in its prompt per activity: a "Gaming" activity hides Spotify search, surfaces Discord push-to-talk; "Work" surfaces Konsole-send and Kate-open, hides music. Implementation is a thin filter at `intent.IntentParser` prompt-build time keyed off the activity ID. Adds a real win for users who already structure work with Activities; no-op for users who don't (single Activity = current behavior). Caveat: the API for "list actions in this activity" needs a per-user mapping, surface it as a `~/.config/korderrc` `[activities]` section.

- **Konsole `sendText` voice action** — Konsole exposes `org.kde.konsole.Session.sendText(string)` over D-Bus on every session object. A `konsole_run` action (parameters: `session=active`, `text=...`) would let voice say *"in terminal, run git status"* or *"in terminal, run docker ps"* without typing-keystroke synthesis. Crucially this avoids the focus-stealing problem of `ydotool` typing into the wrong window. Discovery: enumerate `org.kde.konsole-*` services; pick the focused one via the existing KWin-script bridge (cross-recommendation). Skip dangerous commands (heuristic: refuse `rm -rf`, `sudo`, redirection to `/`) on policy grounds, even though the user is the only operator — tripwires for misheard phrases.

### Cross-app integrations (depends on user's stack)

- **Dolphin file-manager voice navigation** — User wants this if they file-manage with Dolphin. Dolphin doesn't expose a public D-Bus *navigation* interface, but `dolphin --select <path>` and `dolphin <path>` work. KIO `kioclient6 exec <url>` covers protocols (sftp://, fish://, smb://). Voice actions: *"open Downloads in Dolphin"*, *"show /etc/nginx in Dolphin"*. Cheap, KIO-aware (so URLs to remote/cloud volumes Just Work).

- **KeePassXC voice unlock — strongly defer** — KeePassXC exposes `org.keepassxc.KeePassXC.MainWindow` D-Bus methods (`unlockDatabase`, `openDatabase`). Voice would let *"unlock my passwords"* dispatch, with the user's master password coming from a follow-up speech turn or from their KWallet keyring. The security implications of speaking a password are obvious — but unlock-by-voice without speaking the master would still be useful (the master is already cached in KWallet for many users; trigger word fires the unlock). User wants this if KeePassXC is daily-use AND they're OK saying "unlock passwords" out loud. Phase 1 bound: trigger only the unlock dialog, never autofill. Don't touch passwords directly.

- **Obsidian / Logseq / Joplin "append to today's note"** — Obsidian's `obsidian://` URL scheme accepts `obsidian://new?file=2026-05-04&content=...` for note creation. Joplin has a Web Clipper API on `localhost:41184`. Logseq has `logseq://x-callback-url/insertBlock`. Voice action `note_append` (parameters: `app=obsidian|logseq|joplin`, `text=...`) opens or `xdg-open`s the URL. User wants this if they daily-driver a note app. Discovery cost is per-app; Obsidian is the easiest first cut because the URL scheme is documented and stable.

- **KMail / Akonadi voice compose** — KMail exposes `org.kde.kmail.kmail` with `openComposer(to, cc, bcc, subject, body, ...)`. Voice action `email_compose` (parameters: `to`, `subject`, `body`) extracts via Gemma, fires the composer pre-filled. Doesn't *send* — leaves the human in the loop, which fits voice-input failure modes. User wants this if they're on KMail; Thunderbird has the same pattern via `thunderbird -compose` CLI.

- **KOrganizer / Akonadi calendar event** — Akonadi has D-Bus interfaces for calendar resources (`org.kde.korganizer`). Adding *"meeting with Anna tomorrow at 3pm"* requires (a) Gemma extracts {who, when, duration}, (b) call `KOrganizer.openEventEditor(...)` pre-filled. Same human-in-the-loop principle as email; never auto-saves. User wants this if KOrganizer / KMail / Plasma's calendar widget is their actual calendar source.

- **Browser bookmarks via the Plasma Browser Integration extension** — `org.kde.plasma.browser_integration.Bookmarks` exposes the bookmarks tree from the running Firefox/Chromium when the extension is installed. Voice *"open my GitHub bookmark"* fuzzy-matches against the bookmarks list and `xdg-open`s the URL — beats web-search for stable destinations. Implementation: enumerate, fuzzy-match against `args.query`, fall back to the existing `web_search` action when no match. Cost is one new action; user wants this if they actually keep bookmarks.

### Defer or skip

- **AT-SPI for click-by-label (without KWin)** — already parked on `feat/at-spi-and-vision-actions`. AT-SPI enumeration on KDE Wayland is incomplete (some Qt apps not exposing accessibility tree consistently). Don't unpark this on its own — couple it with the **KWin script bridge** above, which provides reliable window enumeration that AT-SPI can supplement with in-window widget enumeration where it works. KWin first; AT-SPI second; Gemma vision only when both fail.

- **Plasma Vault unlock-by-voice** — voice-unlock has the same speak-the-secret problem as KeePassXC. Plasma Vault is also less ubiquitous than the password-manager case, so the user-base reach is narrower. Skip unless the user actually uses Vault daily.

- **KIO general file ops via voice ("save as", "rename")** — these are intrinsically context-bound (the focused app's "Save As" dialog), and the focused-app dialog is the right UX. Voice can't reliably target a dialog that isn't yet open. Stick with `ydotool`-typed filenames for the rare case; don't build a parallel KIO-driven save path that races against the app's own dialog.

- **Khelpcenter voice-search documentation** — Khelpcenter accepts `khelpcenter help:/` URLs but is a small KDE application and not many users live in it. The `Wikipedia` action already covers most "look something up" intents; web search covers the long tail. Cost > value.

- **Plasma "Do Not Disturb" / focus mode toggling** — `qdbus6 org.kde.plasmashell /org/kde/PlasmaShell` and the notification daemon expose Inhibit/Uninhibit. *Tempting* low-hanging fruit, but the activation surface is small (DND for ~30 minutes is already two clicks in the notification widget) and the misfire risk is high (turning DND on by mistake silently breaks notification UX for the day). Skip unless explicitly requested.

- **MPRIS depth: position seek + queue management** — *technically* MPRIS exposes `Seek`, `SetPosition`, `SetTrackList`. *Practically* the player implementations are inconsistent (Spotify Linux client implements ~half of them, Firefox via `plasma-browser-integration` implements fewer still). Voice-controlled seeking is also a low-value command in our experience — the visual scrubber is faster than speech. Skip.

- **MPRIS server while TTS speaks (re-evaluation)** — listed under "low-hanging fruit" above as worth doing. Listing here only to flag the inverse take: *don't* expose Korder as MPRIS for the Whisper-capture phase. Capture isn't "media playback" semantically, and showing it in the media widget alongside Spotify pollutes that surface.

---

## Cross-cutting notes

- **Wayland posture**: every recommendation above is Wayland-clean on Plasma 6. The single piece of X11-era technology Korder still depends on is `ydotool` (uinput), and that is *the* recommended path on Wayland — there is no cleaner replacement for "type arbitrary text into focused app". Don't replace it; just cap how much of Korder's logic depends on keystroke synthesis (the KWin / Konsole / D-Bus / Portal options above all reduce that surface).

- **Single-user scope**: the recommendations stay personal-scope. KDE Connect, KMail/Akonadi, MPRIS server, KWin script — all are first-party-user APIs that don't model "other users". KRunner runner is a possible exception (a multi-user system would expose Korder's voice actions to unintended typists) but the `RunnerV2` protocol is per-session.

- **First-run install footprint**: today Korder installs `.desktop` + hicolor SVG idempotently. The recommendations above add: a systemd unit, optional KWin script (`~/.local/share/kwin/scripts/korder/...`), optional KRunner runner registration. All can fit the same idempotent pattern as `_install_desktop_entry` — write only when absent or content-changed. No package-manager glue needed.
